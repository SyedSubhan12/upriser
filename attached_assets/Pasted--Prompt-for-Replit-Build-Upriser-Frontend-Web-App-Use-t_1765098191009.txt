# Prompt for Replit – Build Upriser Frontend (Web App)

Use this entire document as the **single system prompt** / spec for generating the frontend of the **Upriser Educational Platform**. Your job is to create a **production-ready React + TypeScript frontend** that implements all required layouts, pages, and reusable components described below.

---

## 1. Tech Stack & Project Setup

Build a **single-page web app** with the following stack:

* **Framework:** React 18 with **TypeScript**
* **Bundler:** Vite (React + TS template)
* **Routing:** React Router v6+ (nested routes for roles)
* **Styling:** Tailwind CSS
* **UI Primitives:** Simple, clean, reusable components (Button, Input, Select, Modal, Tabs, etc.).
* **State Management:**

  * React Query (TanStack Query) for server data hooks (use mocked APIs for now)
  * React Context for auth/session + user role
* **Forms:** React Hook Form + Zod (or Yup) for validation

## 2. Global Design System

Create a **simple, modern, school-friendly UI**:

* **Colors:**

  * Primary: `#2563EB` (blue)
  * Primary hover: slightly darker
  * Accent: `#F97316` (orange) for highlights
  * Background: light gray for app shell, white cards
  * Text: near-black for high contrast
* **Typography:**

  * Use a clean sans-serif font (e.g. system font stack)
  * Heading scale: `text-2xl` for page titles, `text-xl` for section headings, `text-base` for body
* **Components:**

  * Buttons: primary, secondary, ghost
  * Inputs: full-width, labels above, helper/error text below
  * Cards: rounded corners, subtle shadow, padding
  * Tables: zebra rows, sticky header, responsive scroll
* **Responsiveness:**

  * All layouts must be fully responsive (desktop, tablet, mobile)
  * Sidebar collapses to a top menu on small screens

Implement this via **Tailwind CSS** utility classes inside the UI components.

## 3. Routing & Layouts

Use **React Router** with nested routes and layouts based on role.

### 3.1 Public Routes

* `/` – `LandingPage` inside `PublicLayout`
* `/login` – `LoginPage`
* `/register` – `RegisterPage` (simple placeholder for now)
* `/select-board` – `BoardSelectionPage` (for initial onboarding)

`PublicLayout` should include:

* Simple header with logo and Login/Register buttons
* Public marketing content area

### 3.2 Auth & Role Handling

* `AuthContext` stores:

  * `user` object with fields `{ id, name, role: 'student' | 'teacher' | 'admin', boards: [...], subjects: [...] }`
  * `isAuthenticated`, `login()`, `logout()`
* Use a **fake login** implementation for now:

  * Email + password form posts to a mocked function in `fakeServer.ts`
  * Return different roles for different test emails (e.g. `student@demo.com`, `teacher@demo.com`, `admin@demo.com`)

`ProtectedRoute` and `RoleGuard` components:

* Redirect unauthenticated users to `/login`
* Restrict access based on role for "/student/*", "/teacher/*", "/admin/*" paths.

### 3.3 Role-Based Layouts

Each role gets a layout with a **top navbar** and **sidebar**.

* `StudentLayout`

  * Sidebar: Dashboard, Study Materials, Practice, Assignments, Announcements, Profile
  * Content area: outlet for nested student pages

* `TeacherLayout`

  * Sidebar: Dashboard, My Materials, Quizzes, Assignments, Class Analytics, Announcements

* `AdminLayout`

  * Sidebar: Dashboard, Boards & Subjects, Users, Content Moderation, Analytics, Settings

Use `AppNavbar`, `AppSidebar`, and `AppShell` to build these layouts.

## 4. Student Pages & Components (MVP)

### 4.1 StudentDashboardPage

Purpose: give a quick overview.

Components used:

* `PageHeader` (title + optional actions)
* `MetricGrid` with `StatCard` items:

  * Total quizzes attempted
  * Average score
  * Active subjects
* Section: "Recent activity" list (last 3 quizzes or materials)

### 4.2 StudyMaterialsPage

Purpose: allow browsing past papers, notes, and videos.

Layout:

* Left: `StudyTreeSidebar` showing Board → Subject → Topic tree
* Right: content area with:

  * `FilterBar` (resource type, year, difficulty)
  * `MaterialList` (grid or list of `ResourceCard`)

Components:

* `StudyTreeSidebar`

  * Props: `boards`, `subjects`, `topics`, callbacks for selection
* `FilterBar`

  * Props: filters (type, year, difficulty) and change handlers
* `ResourceCard`

  * Title, type badge, subject, topic, short description, view/download counts

### 4.3 MaterialDetailPage

* Shows detail for a selected resource:

  * Title, metadata (board, subject, topic, type, year)
  * Description
  * Action buttons: View / Download / Favourite
  * If type is video: embedded player placeholder

### 4.4 QuizPracticePage

Purpose: choose topic and quiz type.

Components:

* `QuizTopicSelector`

  * List of topics with checkboxes or radio buttons
* Simple controls:

  * Number of questions
  * Quiz type (practice vs mock)
  * Timed or not
* Button: "Start Quiz" → navigates to `QuizAttemptPage`

### 4.5 QuizAttemptPage

Purpose: take a quiz.

Layout:

* Top bar: quiz title, `QuizTimer` (if timed), progress (e.g. `Question 3 of 10`)
* Body: `QuizQuestionCard` showing question text and options
* Bottom: Next/Previous and Submit

Components:

* `QuizQuestionCard`

  * Props: question, selected option, onSelect
  * Shows MCQ options with radio-style UI
* `QuizTimer`

  * Props: duration, onTimeUp callback

On submit:

* Show `QuizResultSummary` (in same page or navigate to result page).

### 4.6 QuizHistoryPage

* List of past attempts via `AttemptHistoryList`
* Columns: quiz title, date, score, duration

### 4.7 AssignmentsPage & AssignmentDetailPage

* AssignmentsPage:

  * `AssignmentTable` or simple cards: title, subject, due date, status (pending/submitted/graded)
* AssignmentDetailPage:

  * Details of assignment
  * Upload submission (using `FileUploadField`)
  * Show grade/feedback if already graded

### 4.8 AnnouncementsPage

* Simple list of announcements via `Card`:

  * Title, date, scope (school/board/subject), content snippet

### 4.9 ProfilePage

* Shows student profile info and enrolled boards/subjects
* Simple form to update display name, avatar, and optional fields (not critical to persist yet).

## 7. Shared Components – Behaviour & Props

Implement reusable components with clear props. Some key ones:

### 7.1 Button

```ts
interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'primary' | 'secondary' | 'ghost';
  size?: 'sm' | 'md' | 'lg';
  isLoading?: boolean;
}
```

* Apply Tailwind classes based on variant/size
* Show spinner when `isLoading` is true

### 7.2 Input & Select

* Support `label`, `error`, `helperText` props via wrappers, or combine with `FormField` component.

### 7.3 Modal & ConfirmDialog

* Basic modal with title, content, actions; toggle controlled via props.

### 7.4 DataTable

* Generic table that accepts:

```ts
interface Column<T> {
  key: keyof T | string;
  header: string;
  render?: (row: T) => React.ReactNode;
}

interface DataTableProps<T> {
  columns: Column<T>[];
  data: T[];
}
```

* Used for Users, Materials, Quizzes, Assignments, Submissions.

### 7.5 ToastProvider

* Provide a simple global toast system using context and portals.

## 8. Hooks & API Client (Mocked)

Implement a central `apiClient` and `fakeServer`:

* `apiClient.ts`:

  * Functions like `login`, `getBoards`, `getSubjects`, `getMaterials`, `getQuizzes`, `getAssignments`, etc.
  * For now, delegate to in-memory data in `fakeServer.ts`.

* `fakeServer.ts`:

  * Keep simple arrays of objects for boards, subjects, topics, materials, quizzes, etc.
  * Implement async functions returning `Promise` to simulate network.

Hooks like `useBoards`, `useMaterials`, etc.:

* Wrap React Query calls:

```ts
export function useBoards() {
  return useQuery(['boards'], apiClient.getBoards);
}
```

* Use these hooks in pages instead of calling `apiClient` directly.

## 9. Quality Requirements

* TypeScript **strict** mode enabled; no `any` unless really necessary.
* Components must be **fully typed**.
* Layouts must be **fully responsive** and visually consistent.
* No dead code; keep components small and composable.
* Code style: Prettier + ESLint defaults.
